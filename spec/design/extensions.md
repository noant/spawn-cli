# Extensions

This document describes how to implement an AIDD methodology as a Spawn
extension.

## Terms

Extension is an installable Spawn package that contains one reusable AIDD
methodology slice: files, skills, setup scripts, MCP definitions, and install
metadata.

Extension repository is a repository used to develop one extension. Its
extension source lives under `extsrc/`. Extensions are installed directly from
Extension repositories or repository artifacts after validation; Spawn does not
use a separate package manager for extension distribution.

Build repository is a repository used to publish a bundle of extensions. Its
main contract is `extensions.yaml`, which lists extension sources that can be
installed together with `spawn build install`.

Target repository is the repository where Spawn is initialized and where
extensions are installed under `spawn/`.

Extension source is the `extsrc/` tree inside an Extension repository.

Installed extension is the copied extension source under
`target-repo/spawn/.extend/{ext}/`.

Static file or folder is extension-owned content that may be overwritten during
extension update.

Artifact file or folder is target-owned content created or shaped by an
extension and preserved during update.

Rendered skill is an IDE-specific skill generated by Spawn from extension skill
source and read metadata.

## Purpose

A Spawn extension packages reusable methodology source: compact process files,
skills, optional MCP connections, optional setup scripts, and target repository
files. Installing the extension reproduces the methodology in a target
repository without copying the whole Extension repository itself.

An extension should separate:

- static methodology files that can be updated by the extension;
- artifact files that evolve inside each target repository;
- rendered IDE skills and MCP entries generated by Spawn;
- local target rules owned by repository users.

## Source Layout

An Extension repository uses this source layout:

```text
extend-repo/
  extsrc/
    config.yaml
    mcp.json
    skills/
      {skill}.md
    files/
      {target-path}
    setup/
      {script}.py
```

Required:

- `extsrc/config.yaml`

Optional:

- `extsrc/skills/{skill}.md`
- `extsrc/files/{file}`
- `extsrc/setup/{script}.py`
- `extsrc/mcp.json`

During install, the full Extension source tree is copied into
`target-repo/spawn/.extend/{ext}/` as an Installed extension.
The source is validated before installation. Extension installation does not go
through PyPI, npm, or another package manager.

## Naming

Extension names should be stable, lowercase identifiers, preferably
kebab-case. The name in `extsrc/config.yaml` should match the installed folder
name under `spawn/.extend/{ext}`.

Skill names should also be stable across releases. If a skill name is generic,
prefix it with the methodology or extension name to avoid collisions in IDEs
that store skills in one shared folder.

Target paths declared in `files` and `folders` should be specific enough to
avoid collisions with other extensions. Shared target folders are not part of
the initial contract; prefer extension-specific subfolders unless the
methodology intentionally owns a well-known repository path.

## Config Sections

`extsrc/config.yaml` is the extension contract:

```yaml
version: "1.0.0"
files:
  spec/main.md:
    description: Spec task process.
    mode: static
    globalRead: required
    localRead: required
  spec/design/hla.md:
    description: Project architecture artifact.
    mode: artifact
    globalRead: auto
    localRead: auto
folders:
  spec/tasks:
    mode: artifact
agent-ignore:
  - spawn/.extend/**
git-ignore:
  - .spawn-cache/**
skills:
  spectask-execute.md:
    name: spectask-execute
    description: Execute approved spectasks.
    required-read:
      - spec/tasks/current/overview.md
setup:
  before-install: before_install.py
  after-install: after_install.py
  before-uninstall: before_uninstall.py
  after-uninstall: after_uninstall.py
  healthcheck: healthcheck.py
```

`version` is the extension version.

`files` describes files formed by the extension, either copied from
`extsrc/files/` or created by scripts. Keys are target repository paths.

`folders` describes folders formed by the extension. Keys are target repository
paths.

`agent-ignore` lists globs to add to IDE-specific agent ignore files.

`git-ignore` lists globs to add to the target repository `.gitignore`.

`skills` lists Markdown files under `extsrc/skills/`. A skill may provide
`name`, `description`, and `required-read`. If `name` or `description` is
omitted, Spawn reads it from the skill notation/frontmatter when available.
`required-read` is additive and should contain only files that are mandatory for
that specific skill beyond the files already required by `globalRead` and
`localRead`.

`setup` lists optional idempotent scripts under `extsrc/setup/`.

Unknown fields in `config.yaml` should be treated as validation warnings in
normal mode and validation errors in strict mode. Unknown future schema
versions are errors.

## Static And Artifact Modes

`mode: static` means the extension owns the file or folder. On update, Spawn may
overwrite static files with the new extension version. Static files are suitable
for methodology rules, process descriptions, templates, and other reusable
source that should stay aligned with the extension release.

`mode: artifact` means the target repository owns the file or folder after it
is created. On update, Spawn must preserve artifact content. Artifact paths are
appropriate for task folders, architecture notes, generated project files,
local project state, and other methodology outputs. Artifact files may be
migrated only by explicit setup scripts.

The default mode is `static`.

## Read Rules

`globalRead` controls whether a file appears in the global
`spawn/navigation.yaml` index:

- `required`: read before work in every session.
- `auto`: mention in navigation so the agent can decide whether to read it.
- `no`: do not mention globally.

`localRead` controls whether a file is injected into rendered skills for the
same extension:

- `required`: read whenever a skill from this extension is used.
- `auto`: mention in the skill as contextual.
- `no`: do not mention locally.

The default for both fields is `no`.

Skill-level `required-read` can add mandatory files to one skill without making
those files mandatory for other skills from the same extension. This is used
when a command or workflow needs files that should be read for that skill only,
not for unrelated extension skills.

Skill-level `required-read` should not repeat files that are already included
through `globalRead: required` or `localRead: required`. Spawn merges those
sources automatically and de-duplicates the final rendered skill metadata.

## Skills

Skill source files are plain Markdown under `extsrc/skills/`. They may include
frontmatter with `name` and `description`, but extension config may override
those fields.

Rendered skills are not copied directly from source. Spawn generates
IDE-specific skills from normalized skill metadata:

- resolved name;
- resolved description;
- source body without the optional frontmatter block;
- mandatory reads from skill overrides, local required reads, and global
  required reads;
- contextual reads from local auto reads and global auto reads;
- instruction to consult `spawn/navigation.yaml` for broader repository
  context.

Source skills should stay procedural. Long background explanations usually
belong in extension files with `globalRead` or `localRead` metadata, not in the
skill body itself.

## MCP

`extsrc/mcp.json` defines MCP connections supplied by the extension. Spawn
parses this file into a normalized internal MCP structure, then each IDE adapter
renders that structure into its own MCP configuration format.

Rendered MCP names are stored in
`spawn/.metadata/{ide}/rendered-mcp.yaml` so uninstall and rebuild operations
remove only Spawn-managed MCP entries.

## Setup Scripts

Setup scripts are optional and idempotent:

- `before-install`
- `after-install`
- `before-uninstall`
- `after-uninstall`
- `healthcheck`

Scripts are used for validation, bootstrap, migration, and cleanup that cannot
be expressed as declarative file/folder rules. A failed script should produce a
warning unless the command explicitly treats that phase as blocking.

Scripts execute locally in the target repository and should be considered
trusted code from the extension source. They must not silently delete or
rewrite user-owned files. Destructive behavior should be explicit in the
extension design and limited to Spawn-owned static paths.

Migration scripts should be written so they can infer or receive:

- extension name;
- installed extension path;
- previous installed version when available;
- target version;
- target repository root.

Artifact migrations should be additive or structure-preserving whenever
possible. If a migration cannot be applied safely, it should report a warning
or blocking error rather than overwrite artifact content.

## Methodology Guidance

Files marked as required reads should be compact. Use them for the essence of
the methodology, not for long examples.

Global required reads are strict always-on rules. Keep their count low. Team
language requirements or a small wake-up context are good examples.

Static files should describe reusable rules, processes, and templates. They
should not accumulate project-specific state.

Artifact files are methodology outputs. Architecture documents, project plans,
task folders, and local memory structures should usually be artifacts.

Auto reads let agents select context from the task. Use them for optional
guides, extended references, and domain-specific documents.

Quality checklist for extension authors:

- Required reads are compact and essential.
- Contextual reads have clear descriptions.
- Static files contain reusable methodology source, not target project state.
- Artifact files are preserved across updates.
- Setup scripts are idempotent and safe to rerun.
- MCP definitions do not embed secrets.
- Declared files and folders do not collide with other installed extensions.
- Rendered skill names are stable and unlikely to collide.

## Authoring Workflow

The normal authoring flow is:

1. Create `extsrc/` with `spawn extension init` or by hand.
2. Add static methodology files under `extsrc/files/`.
3. Add source skills under `extsrc/skills/`.
4. Declare files, folders, read rules, skills, ignores, setup scripts, and MCP
   entries in `extsrc/config.yaml`.
5. Run extension validation.
6. Install into a disposable target repository and inspect rendered navigation,
   skills, MCP, and ignores.
7. Run healthcheck.
8. Publish the extension repository or artifact for installation by path, git,
   or zip.

The extension repository should not contain the full target `spawn/` runtime
tree. Only `extsrc/` is copied into target repositories during install.

## Extension From Existing Rules

Spawn can create Extension source from an existing Target repository. The
source target repository may be a local path, git URL with optional branch, or
zip URL. Remote sources are resolved into a temporary local folder before
inspection.

1. Resolve the target repository source when it is git or zip.
2. Read `spawn/rules/` and the current navigation entries from the resolved
   target repository.
3. Classify reusable rules as static extension files.
4. Classify project-specific outputs as artifacts.
5. Copy reusable rules into `extsrc/files/` or `extsrc/skills/`.
6. Write `extsrc/config.yaml` with matching `files`, `skills`, read flags, and
   ignore rules.
7. Keep target-specific history in the target repository instead of moving it
   into extension source.

Generated extension source should be reviewed before publishing. Existing
rules often mix reusable methodology with local team decisions, so authors
should decide which files belong in the extension and which should remain
target-local `spawn/rules/`.

## Build Extensions

A Build repository is a methodology bundle: a repository that lists multiple
extensions so a team can install a complete AIDD methodology set with one
command. It lists extensions from other repositories or artifacts; it does not
contain those extension sources by itself. It does not have to be an extension
itself; its main contract is the `extensions.yaml` manifest.

A Build repository may contain `extensions.yaml`:

```yaml
extensions:
  - path: https://example.com/org/spectask.git
    branch: main
  - path: https://example.com/org/team-rules.zip
```

`spawn build install` resolves each entry, installs or updates the listed
extensions, and performs a final rebuild of navigation, skills, MCP, and ignore
files.

Build repositories allow teams to publish a complete AIDD methodology as a set
of smaller extensions and install that full set in a target repository with a
single `spawn build install` command.

In this design, "build extension" means a build repository or build manifest
that composes multiple extensions. It is not itself required to be an
installable extension unless it also contains a separate `extsrc/` tree.

## Minimal Extension Example

```text
extend-repo/
  extsrc/
    config.yaml
    files/
      methodology/guide.md
    skills/
      methodology-work.md
```

```yaml
name: methodology
schema: 1
version: "1.0.0"
files:
  methodology/guide.md:
    description: Core methodology guide.
    mode: static
    localRead: required
skills:
  methodology-work.md:
    name: methodology-work
    description: Apply the methodology to a task.
```

This extension installs one static guide and one rendered skill. The guide is
required only when using skills from the same extension, not globally for every
agent task.

## Publishing

Publishing in this document refers to publishing the `spawn-cli` utility itself,
not publishing extensions. The CLI package is intended to be published to PyPI,
for example so users can run it as `uvx spawn-cli ...`.

Extensions are installed directly from Extension repositories or repository
artifacts after validation. Future CLI publishing can reuse the existing
publish-script pattern from the related spectask CLI work, with the package name
adapted to `spawn-cli`.
